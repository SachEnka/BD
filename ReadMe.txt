1. Цель работы
научиться создавать приложения, использующие средства потоковой обработки данных в реальном времени.

2. Используемые технологии
Python 3.x - язык программирования
Apache Kafka - платформа для потоковой передачи данных
kafka-python - библиотека для взаимодействия с Kafka
JSON - формат обмена данными

3. Структура проекта
Проект состоит из следующих файлов:
consumer.py - основной файл потребителя, содержащий логику подключения и обработки сообщений
config.py - файл конфигурации с параметрами подключения к Kafka
validator.py - модуль валидации, проверяющий корректность входящих сообщений
requirements.txt - файл со списком зависимостей проекта
.venv/ - директория виртуального окружения

4. Описание компонентов
4.1 Модуль конфигурации
Хранит параметры подключения к Kafka, включая адрес сервера и название топика. Это позволяет централизованно управлять настройками и легко изменять их без модификации основного кода.

4.2 Модуль валидации
Выполняет проверку входящих сообщений на соответствие требуемому формату. Основные функции:
Проверка на пустое сообщение
Парсинг JSON структуры
Проверка наличия обязательных полей
Валидация типов данных
Возврат результата проверки и сообщения об ошибке

4.3 Основной модуль потребителя
Реализует логику работы consumer:
Создание подключения к Kafka с указанием параметров
Бесконечный цикл ожидания и обработки сообщений
Вызов модуля валидации для каждого сообщения
Вывод результатов обработки в консоль
Сбор статистики по обработанным сообщениям

4.4 Файл зависимостей
Содержит единственную зависимость - библиотеку kafka-python версии 2.0.2, необходимую для работы с Kafka из Python.

5. Установка и настройка
5.1 Подготовка окружения
Для работы проекта необходимо создать и активировать виртуальное окружение Python, затем установить зависимости из файла requirements.txt с помощью менеджера пакетов pip.

5.2 Настройка Kafka
Перед запуском consumer необходимо запустить Kafka брокер. Для этого требуется:
Запустить Zookeeper, который используется Kafka для управления метаданными
Запустить Kafka сервер, который принимает и обрабатывает сообщения
Создать топик с указанием количества партиций и фактора репликации
5.3 Запуск потребителя
Consumer запускается как обычный Python скрипт. После запуска он подключается к Kafka и начинает ожидать сообщения в указанном топике.

6. Тестирование
6.1 Сценарии тестирования
Для проверки работы consumer были использованы следующие типы сообщений:
Валидное сообщение - содержит все обязательные поля с корректными типами данных
Сообщение с отсутствующим обязательным полем
Сообщение с неверным типом данных в одном из полей
Сообщение с некорректной JSON структурой

6.2 Результаты тестирования
При обработке тестовых сообщений consumer демонстрирует следующее поведение:
Валидные сообщения помечаются как успешно прошедшие проверку
Сообщения с пропущенными полями отклоняются с указанием конкретного отсутствующего поля
Сообщения с неверными типами данных отклоняются с указанием поля и ожидаемого типа
Сообщения с синтаксическими ошибками JSON отклоняются с описанием ошибки парсинга
Для каждого сообщения выводится дополнительная информация: смещение в партиции, номер партиции и временная метка.
По завершении работы выводится статистика: общее количество сообщений, количество валидных и невалидных сообщений, процент успешной валидации.

7. Возможные ошибки и их решение
Ошибка	Причина	Решение
ImportError,	Отсутствует библиотека kafka-python,	Установить пакет через pip
NoBrokersAvailable,	Kafka сервер не запущен,	Запустить Kafka и Zookeeper
Topic not found	,Топик не создан в Kafka,	Создать топик с помощью утилиты kafka-topics
Connection refused,	Неверные параметры подключения,	Проверить адрес и порт в config.py
8. Архитектура системы
Система построена по модели "издатель-подписчик". Producer отправляет сообщения в топик Kafka, который выступает в роли буфера. Consumer подключается к этому топику и получает сообщения в порядке их поступления.
После получения сообщение передается в модуль валидации, который проверяет его структуру и содержимое. Результат валидации вместе с метаданными сообщения выводится в консоль.
Такая архитектура обеспечивает слабую связанность компонентов и позволяет масштабировать систему путем добавления дополнительных потребителей в одну группу.
